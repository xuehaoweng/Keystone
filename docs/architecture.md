# LLM Gateway 架构说明

## 定位

LLM Gateway 是企业内部统一的大模型入口。它不是简单反向代理，而是把多模型接入、鉴权、限流、配额、成本、路由、容灾和审计集中到一个治理面。

典型问题包括：

- 多个业务线分别直连不同模型，调用方式和安全策略不一致。
- API Key 被多人共用，无法区分实习生测试和核心业务调用。
- 简单任务使用高价模型，复杂任务误用轻量模型，成本和质量同时失控。
- 某个模型供应商限流或宕机时，业务侧没有统一降级能力。
- 财务和平台团队看不到谁用了多少 token、花了多少钱。

## 三层架构

```text
业务系统
  ↓
接入治理层：鉴权、限流、白名单、quota、审计
  ↓
智能路由层：场景、成本、延迟、模型状态、负载均衡
  ↓
模型执行层：协议转换、重试、降级、熔断、缓存
  ↓
OpenAI / Claude / Qwen / DeepSeek / Kimi / GLM / 私有模型
```

### 1. 接入治理层

面向调用方，回答“谁能进来、能进来多少、能使用什么能力”。

当前实现：

- API Key 和 JWT 鉴权。
- 按用户维度 QPS 限流。
- API Key 月度 token quota。
- API Key `allowed_tiers` 权限控制。
- API Key 创建、列表、删除和用量查询。

后续可前置到 Kong、Nginx、APISIX 或 Envoy：

- IP 白名单。
- 企业 SSO。
- 更细粒度 RBAC。
- 统一审计和 WAF 策略。

### 2. 智能路由层

面向策略，回答“这个请求应该使用哪类模型、哪个模型实例”。

当前实现：

- 显式 `model` 最高优先级。
- `preferred_model` 优先模型。
- `model_tier=cheap|expensive` 指定模型层级。
- `model_tier=auto` 时先走静态规则，再走意图分类。
- 同 tier 内按健康状态、连接数和权重选择模型。
- 非流式请求失败后尝试同 tier 其他模型，再尝试另一个 tier。

路由策略可以继续扩展为：

- 成本优先。
- 质量优先。
- 延迟优先。
- 业务线专属模型池。
- 灰度和 A/B。
- 按模型实时错误率动态降权。

### 3. 模型执行层

面向下游供应商，回答“如何稳定调用不同模型并抹平差异”。

当前实现：

- 多 provider adapter。
- 统一请求和响应模型。
- 流式和非流式响应。
- 失败计数、熔断冷却和自动恢复。
- 结果缓存服务，适合低温度确定性请求。
- token usage 和成本估算持久化。

后续可以继续增强：

- provider API Key 轮换。
- 更细粒度错误归一化。
- 半开探测请求。
- Redis 分布式负载状态。
- 结果缓存命中率、节省成本统计。

## 数据面和控制面

数据面是 `/api/runs`，承载真实模型调用。

控制面包括：

- `/api/auth/keys`
- `/api/auth/keys/{key_id}/usage`
- `/api/models`
- `/api/models/{model_name}/health`
- `/api/metrics`
- `/api/config`
- `/admin`

控制面必须比数据面更严格地保护权限。当前版本所有控制面接口都要求鉴权，后续应区分 admin、operator、viewer 和 app key。
