# 项目开发约束

## 工作流程（必须严格遵守）

1. **先了解需求** - 仔细阅读用户需求，不明确时先询问澄清
2. **输出解决方案** - 提供实现方案供用户审核
3. **等待审核** - 必须获得用户明确批准后才能执行
4. **审核通过后再执行代码修改** - 禁止未经审核直接实现
5. **记录变更** - 修改后更新 `.claude/CHANGES.md`

## 禁止行为

- ❌ 禁止未经审核直接实现代码
- ❌ 禁止紧张实现方案之外的需求（不做额外"优化"）
- ❌ 禁止"顺手"修改操作范围之外的代码
- ❌ 禁止让用户自己修改代码
- ❌ 禁止跳过方案审核步骤

## 大需求处理

- 复杂需求必须拆分为可独立验证的子任务
- 每个子任务完成后需等待用户确认再进行下一步
- 使用任务列表追踪进度

## 代码修改规范

- 每次修改前说明修改内容和影响范围
- 修改后必须进行自测（语法检查、导入测试）
- 保持修改最小化，不引入无关变更

## 配置文件保护（Hook 拦截）

以下文件**禁止直接修改**，修改前必须获得用户明确批准：

### 受保护的文件列表

| 文件 | 说明 |
|------|------|
| `config.py` | 主配置文件 |
| `config.example.py` | 配置示例文件 |
| `.env` / `.env.*` | 环境变量文件 |
| `docker-compose.yml` | Docker 编排文件 |
| `dockerfiles/*` | Docker 构建文件 |
| `requirements.txt` | Python 依赖文件 |
| `.gitignore` | Git 忽略规则 |
| `.git/*` | Git 配置文件 |
| `.claude/settings.json` | Claude 设置文件 |
| `.claude/settings.local.json` | Claude 本地设置文件 |

### Hook 机制

- 修改上述文件前，会自动触发 `.claude/hooks/protect-config.sh` 脚本
- Hook 会拦截未授权的修改请求并输出警告
- 用户明确批准后，才能临时解除保护
